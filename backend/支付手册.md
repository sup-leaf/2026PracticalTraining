# 校园集市 - 支付模块手册

> 基于虎皮椒（Xunhupay）支付网关
> 更新时间：2026-06-08

---

## 一、模块概述

支付模块采用**双模式**架构。

| 模式 | 配置值 | 行为 |
|------|--------|------|
| **simulate** | `hpj.mode=simulate` | 不调用真实 API，confirm=1 直接标记支付成功 |
| **production** | `hpj.mode=production` | 直连虎皮椒网关，发送 form-urlencoded 请求 |

## 四、API 接口

| 端点 | 方法 | 说明 |
|------|------|------|
| `/api/payment/vip/create` | POST | 创建 VIP 支付订单（￥0.02） |
| `/api/payment/order/query?orderNo=` | GET | 查订单状态 |
| `/api/payment/order/{orderNo}` | DELETE | 取消 pending 订单 |
| `/api/payment/notify` | POST | 虎皮椒异步回调（预留） |

### 创建订单响应（生产模式）

```json
{
  "orderNo": "PAY...",
  "amount": "0.02",
  "mode": "production",
  "payUrl": "https://api.xunhupay.com/payments/wechat/index?id=...",
  "qrcode": "https://api.xunhupay.com/payments/wechat/qrcode_v3?id=..."
}
```

### 查单响应

```json
{ "orderNo": "PAY...", "status": "success", "tradeNo": "420000...", "memberLevel": 1 }
```

## 五、真实支付流程

```
1. 前端 POST /api/payment/vip/create
   → 后端调虎皮椒 /payment/do.html（form-urlencoded + MD5 签名）
   → 返回 payUrl（微信支付页面链接）
   → 创建 PaymentOrder(status=pending)

2. 前端打开 payUrl（新窗口），用户扫码支付

3. 用户点击"我已支付" → GET /api/payment/order/query?orderNo=xxx
   → 后端调虎皮椒 /payment/query.html（查单 API）
   → status=OD → 更新 PaymentOrder.success
   → 升级 User.memberLevel=1
   → 前端显示"已升级VIP"
```

## 六、签名算法

```
1. 收集所有参数，过滤空值和 hash 字段
2. 按键 ASCII 码排序
3. 拼接 key1=val1&key2=val2...
4. 末尾直接追加 appSecret（无 &key= 分隔）
5. MD5 转小写
```

实现位置：`HpjPayUtil.java:generateSign()`

## 七、虎皮椒 API 参数对照

| 操作 | API | 订单号参数名 | 成功判断 | 返回字段 |
|------|-----|-------------|---------|---------|
| 下单 | `/payment/do.html` | `trade_order_id` | `errcode=0` | `url`, `url_qrcode`, `openid` |
| 查单 | `/payment/query.html` | `out_trade_order` | `errcode=0` | `data.status`(OD/WP), `data.transaction_id` |

## 九、已验证的 Bug 修复记录

| # | 问题 | 根因 | 修复 |
|:--:|------|------|------|
| 1 | `payUrl` 始终为空 | `RestTemplate` 默认发 `application/json`，虎皮椒要求 `application/x-www-form-urlencoded` | `buildFormBody()` 拼参数 + `HttpHeaders` 设 Content-Type |
| 2 | `errcode=0` 被误判失败 | 判断条件写成 `"200".equals(errcode)` | 改为 `isOk()` 兼容 `Number 0` 和 `"0"` |
| 3 | 签名错误（中文编码） | `buildFormBody` 未对参数值做 URL 编码 | 加入 `URLEncoder.encode(value, UTF_8)` |
| 4 | 签名错误（空值差异） | `buildFormBody` 把 `return_url=""` 发出去，但 `generateSign` 过滤了空值 | `buildFormBody` 与 `generateSign` 统一过滤空值 |
| 5 | 查单返回 `Invalid order` | 查单参数名用了 `trade_order_id`（下单的参数），应为 `out_trade_order` | `trade_order_id` → `out_trade_order` |
| 6 | 查单成功但解析不出状态 | 返回字段名为 `data.status` 和 `data.transaction_id`，代码读的是 `trade_status` 和 `trade_no` | `trade_status` → `status`，`trade_no` → `transaction_id` |
| 7 | 模拟确认代码被注释 | backk 同步时模拟支付 if 块意外被注释 | 取消注释 |
| 8 | pending 订单无法清理 | 旧订单卡 pending 时新建被拒绝 | `createVipOrder` 改为自动删除旧 pending 订单 |

## 十、切换模拟/真实模式

**模拟模式**：`application-hpj.properties` 中 `hpj.mode=simulate`

**真实模式**：`hpj.mode=production` 并注释 `PaymentController.queryOrder()` 中模拟确认代码块（已默认注释）。

## 十一、前端对接（Profile.tsx）

```typescript
// 升级VIP
const toggleVip = async () => {
  // 1. 创建订单
  const { orderNo, payUrl, mode } = (await api.post('/payment/vip/create')).data;
  // 2. 模拟模式直接确认；生产模式打开支付页
  if (mode === 'simulate') {
    await api.get(`/payment/order/query?orderNo=${orderNo}&confirm=1`);
    // 更新 memberLevel
  } else {
    window.open(payUrl, '_blank');
    // 用户支付后点击"查询状态"按钮
  }
};

// 查询支付状态
const checkPayStatus = async () => {
  const { status } = (await api.get(`/payment/order/query?orderNo=${orderNo}`)).data;
  if (status === 'success') { /* 升级成功 */ }
};
```
